# Ecom Strict Implementation Plan

## Goal

Deliver a local-first React workbench where a user can create a product project, upload and restore reference assets, plan Taobao or Amazon delivery slots, generate and version slot images through demo or OpenAI-compatible engines, review platform-specific compliance reminders, and export a platform package.

## Background And Constraints

- The visual direction is a narrow dark platform rail with a bright operational work surface.
- The MVP platforms are Taobao / Tmall and Amazon.
- Platform rules must be independent rule packs rather than UI constants.
- Demo mode must complete the workflow without paid API calls and must be visibly labeled.
- Real API support is best-effort and depends on provider CORS, endpoint, model, quota, and image capability.
- Accounts, cloud sync, publishing, and automatic visual compliance are out of scope.
- Xiaobai design-system and experience-acceptance rules are mandatory gates.

## Architecture

Use React, TypeScript, Vite, Zustand, native IndexedDB, and fflate. Domain modules own projects, assets, platform packs, planning, generation, compliance, versions, tasks, and export. UI components consume those interfaces and never hard-code platform-specific slots. Demo and OpenAI-compatible engines implement the same planner and image-generator interfaces.

## Global Verification

```bash
pnpm test
pnpm run typecheck
pnpm run build
pnpm run dev
```

Final acceptance additionally requires real browser evidence at desktop and narrow-screen viewports, core workflow interaction, console/overflow checks, and the Xiaobai coverage matrix.

## Task 1: A user can open the responsive workbench shell

Goal:
Render the dark platform navigation, bright context band, workspace regions, settings dialog entry, and mobile bottom navigation from shared design tokens.

Files:
- `package.json`, Vite/TypeScript configuration
- `src/main.tsx`, `src/App.tsx`, `src/styles.css`
- `src/components/AppShell.tsx`, `PlatformRail.tsx`, shared primitives
- `src/domain/platforms/types.ts`

Interfaces:
- Input: navigation item definitions and active destination.
- Output: responsive shell that preserves the same task priority on desktop and narrow screens.

Test:
- Add a design-contract unit test for active navigation definitions and supported platform IDs.
- Add a browser smoke that checks shell regions and no horizontal overflow.

Expected first failure:
- Tests fail because platform definitions and rendered shell do not exist.

Implementation:
- Scaffold the app and token layer.
- Implement shared button, badge, field, dialog, tooltip, and status primitives.
- Implement responsive navigation and context band.

Verification:
- `pnpm test -- platform`
- `pnpm run typecheck`
- Browser shell smoke.

Completion evidence:
- Desktop and narrow shell render with supported real destinations only.

Failure handling:
- Enter `diagnose`; do not add page-specific CSS patches before locating the shared contract failure.

## Task 2: A user can create and restore a product project with assets

Goal:
Create a project, edit product facts, upload compressed reference images, reload, and restore the same project and thumbnails.

Files:
- `src/domain/projects/*`
- `src/domain/assets/*`
- `src/store/workbench-store.ts`
- `src/components/ProjectDialog.tsx`, `ProductSourcePanel.tsx`, `AssetLibrary.tsx`

Interfaces:
- `ProjectRepository` for metadata persistence.
- `AssetRepository` for Blob storage and retrieval.
- Stable IDs generated by `crypto.randomUUID()` with fallback.

Test:
- Project create/update/restore behavior using memory repositories.
- Asset metadata merge and delete behavior.

Expected first failure:
- Tests cannot create repositories or restore a stable active project.

Implementation:
- Add localStorage project persistence and native IndexedDB asset persistence.
- Compress large uploads before storage.
- Revoke object URLs when replaced or unmounted.

Verification:
- `pnpm test -- projects assets`
- Manual refresh restores project facts and asset thumbnails.

Completion evidence:
- Refresh does not lose the active project or uploaded image.

Failure handling:
- Enter `diagnose`; isolate repository initialization and browser storage failures.

Test:

- Amazon MAIN and A+ rules.
- Demo planner completeness.
- Malformed API payload normalization and visible failure.

Expected first failure:
- Tests fail because rule packs, planner interface, and slot normalization are absent.

Implementation:
- Write independent rule packs.
- Implement deterministic demo planning.
- Implement OpenAI-compatible Chat Completions planning with abort and timeout.
- Add loading lock, retry, and stale-response guards.

Verification:
- `pnpm test -- platforms planning`

Completion evidence:
- Both platforms produce distinct complete slot boards and preserve input on failure.

Failure handling:
- Enter `diagnose`; distinguish schema, network, auth, model, and stale-selection errors.

## Task 4: A user can generate, edit, regenerate, and switch versions

Goal:
Generate the selected slot, edit its prompt, regenerate without overwriting history, and activate an earlier version.

Files:
- `src/domain/generation/*`
- `src/services/demo-image-generator.ts`, `openai-image-generator.ts`
- `src/components/GenerationActions.tsx`, `VersionStrip.tsx`, result media components

Interfaces:
- `ImageGenerator.generate(request, signal)`
- Immutable `SlotVersion`
- Repository operations to add and activate versions.

Test:
- Demo generation creates a labeled deterministic image.
- Successful generation appends a version and activates it.
- Failure leaves prior versions untouched and exposes retry.
- Activation changes only the active version ID.

Expected first failure:
- Tests fail because generation interfaces and version behavior do not exist.

Implementation:
- Add SVG/bitmap demo generation.
- Add OpenAI-compatible generations/edits parsing for base64 or URL output.
- Persist generated Blobs in IndexedDB.
- Add pending locks, abort, retry, prompt editing, and version selection.

Verification:
- `pnpm test -- generation versions`
- Browser slow/pending, success, failure/retry, and version switch scenarios.

Completion evidence:
- A slot visibly retains at least two switchable versions after regeneration.

Failure handling:
- Enter `diagnose`; preserve previous active output while investigating.
